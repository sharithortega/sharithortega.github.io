Pruebas de software

Introducción

Las pruebas son una actividad destinada a evaluar y mejorar la calidad de los productos e identificar defectos y problemas.

Las pruebas de software consisten en comprobar dinámicamente el comportamiento de un programa con un conjunto limitado de casos de prueba seleccionados en consecuencia de un número normalmente infinito de ejecuciones frente al comportamiento esperado. En la definición anterior las palabras en cursiva corresponden a los aspectos relevantes en la definición de "Competencia Procesal". En particular:

Dinámico

Este término significa que las pruebas siempre implican ejecutar un programa con datos de entrada (evaluados). Para ser claros no siempre es suficiente ingresar valores para definir una prueba porque un sistema complejo y no determinista puede comportarse de manera diferente con la misma entrada de datos dependiendo del estado en el que se encuentre. En cualquier caso conservamos el término "entrada de datos" en esta KA asumiendo que el término incluye un estado específico del sistema cuando corresponda. Técnicas de prueba adicionales aunque diferentes se describen en Software Quality Assurance KA.

Limitado

En teoría incluso los programas simples pueden tener tantas pruebas que las pruebas exhaustivas pueden llevar meses o años. Por tanto prácticamente todo el conjunto de pruebas podría considerarse infinito. Las pruebas siempre implican una compensación entre recursos y horarios de trabajo limitados por un lado y las necesidades inherentes de pruebas ilimitadas por el otro.

Seleccionado

Una diferencia importante entre las técnicas de prueba propuestas es cómo se selecciona el conjunto de prueba. Los ingenieros informáticos deben ser conscientes de que diferentes criterios de selección pueden producir eficiencias muy diferentes. Identificar los criterios de selección de pruebas más apropiados para un conjunto dado de condiciones es un problema complejo; En la práctica se utiliza la experiencia en planificación de pruebas y técnicas de análisis de riesgos.

Esperado

Debería ser posible aunque a veces no fácil decidir si el resultado observado al ejecutar un programa es aceptable o no; de lo contrario las pruebas serían inútiles. El comportamiento observado se puede comparar con los resultados esperados del usuario (comúnmente conocidos como pruebas de validación) con especificaciones (pruebas de confirmación) o finalmente con el comportamiento predicho por requisitos implícitos o expectativas razonables. Consulte Requisitos de software TAMBIÉN la Sección 6.4 Pruebas de aceptación para obtener más información.

Realizar pruebas para la identificación de defectos

Cuando realizamos pruebas para la identificación de defectos una prueba es satisfactoria si produce un error en el sistema. Es éste un enfoque completamente diferente al de realizar pruebas para demostrar que el software satisface las especificaciones u otro conjunto de propiedades deseadas en cuyo caso una prueba satisfactoria es aquella en la que no se observan errores (al menos significativos).

El problema del oráculo

Un oráculo es cualquier agente (humano o mecánico) que decide si un programa funciona correctamente durante una prueba produciendo así un resultado de "aprobado" o "reprobado". Hay muchos tipos de oráculos y la automatización de oráculos puede resultar muy compleja y costosa.

¿Qué son las pruebas de software?

Las pruebas de software son un conjunto de actividades y procesos cuyo objetivo es evaluar la calidad del software y detectar errores antes de su entrega a los usuarios finales. Es una parte crucial de la gestión de proyectos de software ya que ayuda a garantizar que el software desarrollado cumpla con los requisitos establecidos sea confiable seguro y funcione correctamente.

¿En qué consisten las pruebas de software?

Las pruebas de software involucran diversas actividades entre las que destacan:

  • Planificación de pruebas: Define el alca nce de las pruebas, los objetivos, las estrategias, el calendario y los recursos necesarios para llevarlas a cabo. Se establecen los casos de prueba, los criterios de aceptación y las herramientas de prueba que se utilizarán.
  • Diseño de pruebas: Crea casos de prueba detallados que describen las acciones a realizar, los datos de entrada esperados y los resultados esperados para cada prueba. Se diseñan diferentes tipos de pruebas como pruebas unitarias, pruebas de integración, pruebas de sistemas y pruebas de aceptación.
  • Implementación de pruebas: Desarrolla y ejecuta los casos de prueba diseñados utilizando herramientas de prueba automatizadas o manuales. Se registran los resultados de las pruebas, los errores encontrados y las incidencias reportadas.
  • Análisis de pruebas: Evalúa los resultados de las pruebas para identificar la causa de los errores encontrados y determinar si el software cumple con los criterios de aceptación. Se analiza la cobertura de las pruebas y se prioriza la corrección de errores de acuerdo a su severidad.
  • Informes de pruebas: Documenta los resultados de las pruebas, los errores encontrados, las acciones de corrección realizadas y el estado actual del software. Se comunican los resultados a las partes interesadas del proyecto y se proporciona información para la toma de decisiones.

¿Por qué son importantes las pruebas de software?

Las pruebas de software son esenciales por varias razones:

  • Detectan errores: Las pruebas ayudan a identificar y corregir errores en el software antes de que llegue a los usuarios finales. Esto evita problemas, fallos y frustraciones para los usuarios y protege la reputación del desarrollador.
  • Garantizan la calidad: Las pruebas ayudan a asegurar que el software cumpla con los requisitos establecidos y funcione de la manera esperada. Esto proporciona confianza a los usuarios y a las partes interesadas en el proyecto.
  • Mejoran la confiabilidad: Las pruebas ayudan a reducir la probabilidad de que el software falle o se comporte de manera inesperada. Esto aumenta la confiabilidad del software y lo hace más seguro para su uso.
  • Reducen costos: Es más barato corregir errores en las primeras etapas del desarrollo que después de que el software ha sido lanzado. Las pruebas ayudan a detectar errores tempranos y reducir los costos de corrección a largo plazo.

Tipos de pruebas de software

Existen diferentes tipos de pruebas de software que se enfocan en distintos aspectos del software:

  • Pruebas unitarias: Prueban módulos o componentes individuales del software de forma aislada.
  • Pruebas de integración: Verifican que los diferentes componentes del software se comuniquen y funcionen correctamente en conjunto.
  • Pruebas de sistemas: Evalúan el funcionamiento general del software como un sistema completo.
  • Pruebas de aceptación: Determinan si el software cumple con los requisitos establecidos por los usuarios y las partes interesadas.
  • Pruebas de rendimiento: Miden el rendimiento del software bajo carga y evalúan su capacidad para manejar un número determinado de usuarios o transacciones.
  • Pruebas de seguridad: Verifican que el software esté protegido contra ataques y vulnerabilidades y que los datos de los usuarios estén seguros.

Al realizar pruebas exhaustivas se pueden detectar y corregir errores antes de que el software llegue a los usuarios finales lo que garantiza un producto confiable seguro y funcional.
